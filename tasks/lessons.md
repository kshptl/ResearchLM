# Lessons Learned

- 2026-03-02: When user feedback says front-end planning is lacking, expand planning artifacts with explicit UI interaction contracts (layout, canvas capabilities, node taxonomy, undo/redo depth, and platform scope) before regenerating tasks.
- 2026-03-04: When both local installs and an upstream open-source repo exist, prioritize upstream source code as the primary truth for behavior parity mapping and use local binaries only as secondary verification.
- 2026-03-04: Before stating a fix works, always run verification commands first and report concrete results (at minimum targeted tests plus project test/typecheck if available).
- 2026-03-04: Model/provider selection UX must exist at the initial generation entry point, not only in post-generation detail panels, so the very first request is explicitly user-controlled.
- 2026-03-04: Before pushing, confirm and use the exact target branch requested by the user (for example `main`) and avoid pushing feature branches unless explicitly asked.
- 2026-03-04: When access depends on configured credentials, absence of credentials must have an explicit in-product guidance state (toast/banner with direct path to auth settings), not just hidden controls.
- 2026-03-04: Auth/provider UI tests alone are insufficient; for OAuth provider paths, add adapter-level request-shape regression tests (required fields like `instructions`) so first-prompt live generation failures are caught before release.
- 2026-03-04: For OpenAI OAuth/Codex integrations, treat request-shape defaults from OpenCode (`instructions`, `store: false`) as mandatory parity checks and lock them with adapter-level tests.
- 2026-03-04: Add at least one end-to-end generation-path test per OAuth provider path (not just settings/auth UI tests) so provider-required payload flags are validated before release.
- 2026-03-04: When user wants manual output validation, remove heuristic quality-guard code paths and stale `quality:` UI classifications together; leaving either one causes confusing false-positive notifications.
- 2026-03-04: Any regenerate action must reuse the same prompt-context builder as initial submit; if these diverge, child nodes lose ancestor context after parent updates.
- 2026-03-04: Prompt editing UX requires explicit keyboard semantics (`Enter` submit, `Shift+Enter` newline) and should be enforced with interaction tests, not only visual/manual checks.
- 2026-03-04: In the side panel, keep prompt as the editable source of truth and response read-only; add context actions (like follow-up) on selected response text rather than making response directly editable.
- 2026-03-04: Wheel behavior on canvas nodes should be focus-aware: consume wheel events over focused node content to prioritize local scroll, but let non-hover wheel events bubble so canvas zoom remains available.
- 2026-03-04: For rounded node designs, default square resize frames can break visual consistency; prefer transparent resize lines with rounded handle styling so controls match node geometry.
- 2026-03-04: When a file edit is needed, use the dedicated `apply_patch` tool (or direct file writes) instead of invoking `apply_patch` through `exec_command`.
- 2026-03-04: When converting canvas controls to node-level actions, keep accessibility names distinct from side-panel/context-menu actions to avoid ambiguous test and assistive-technology selection.
- 2026-03-04: For node action-dense UI, use shadcn card sections (`CardHeader`/`CardContent`/`CardFooter`) with icon-first footer actions and hover-revealed labels to preserve compactness without losing discoverability.
- 2026-03-04: Side-panel UX must preserve parallel readability: settings should behave as a floating sibling panel (not overlaying node detail), and inline node prompt editing should always exit on click-away to avoid sticky edit states.
- 2026-03-04: For resizable graph nodes, keep scrolling scoped to the content section and pin footer actions with a non-shrinking layout; avoid whole-card scroll that hides node actions.
- 2026-03-04: For icon-only node actions, expose labels via shadcn tooltips rather than inline hover text inside buttons; keep ARIA labels stable for accessibility and tests.
- 2026-03-04: When migrating to shadcn tooltips, remove all leftover native `title` attributes from buttons (including side-panel utility buttons), otherwise browser-default tooltips create inconsistent UX.
- 2026-03-04: If a shadcn component appears visually wrong despite correct JSX usage, verify Tailwind semantic token mappings (`popover`, `accent`, `muted`, etc.) and root CSS vars; missing token mappings can make components look transparent even when component wiring is correct.
- 2026-03-04: When matching a visual reference, validate the shared primitive style contract itself (not only usage sites); for tooltip parity this includes container tone, text contrast, arrow pointer, and animation semantics.
- 2026-03-04: When product scope removes a feature group (toolbar modes/history/hierarchy), remove both the visible controls and the backing state/effects/keyboard handlers, then update UI contract tests so behavior and tests stay aligned.
- 2026-03-04: For shadcn/Radix motion parity, verify Tailwind plugin wiring (`tailwindcss-animate`) in addition to component JSX; missing plugin wiring silently disables all `animate-in/out` utility behavior.
- 2026-03-04: When user-corrected visual parity conflicts with prior assumptions, prefer the canonical shadcn default primitive styling (avoid over-custom seams/borders unless explicitly required).
- 2026-03-04: For shadcn component parity requests (like combobox), use the documented recipe structure directly (`Button + Popover + Command`) and avoid ad-hoc custom abstractions/styles that diverge from docs defaults.
- 2026-03-04: On Tailwind v4 migrations for shadcn projects, verify the complete stack together: `tailwindcss` + `@tailwindcss/postcss` + CSS-first globals + `tw-animate-css` + `components.json` v4 config shape; partial migrations cause subtle UI regressions.
- 2026-03-04: When shadcn tooling hints a required app-level provider (for example `TooltipProvider`), wire it at root layout and remove local ad-hoc provider overrides unless there is a concrete UX reason.
- 2026-03-04: When users request "no shadcn customizations," re-sync with `shadcn add --all --overwrite`, then remove conflicting legacy theme blocks (`hsl(var(--...))` vs v4 token vars) so generated components render as-designed.
- 2026-03-04: For cluttered auth settings, collapse method sprawl into provider-specific tabs and replace free-form credential management panels with a simple saved-credentials table + masked secrets + revoke-only action.
- 2026-03-04: Conflict notices should not auto-open from historical persisted conflict events on startup; only newly detected conflicts in the current session should surface as active notices.
- 2026-03-04: For shadcn Sonner parity, keep neutral defaults (`<Toaster />` without `richColors`), set a consistent top-entry position unless explicitly requested otherwise, and keep dismiss affordance enabled globally.
- 2026-03-04: Keep generation-failure toasts user-facing and concise; do not append internal classification/action metadata (for example `unknown • Actions: ...`) in the visible toast description.
- 2026-03-04: On Tailwind v4 with generated shadcn Card slots, watch for implicit spacing utilities (`gap`, `py`, conditional header/footer paddings). For dense node UIs, explicitly neutralize slot spacing (`gap-0 py-0`) and avoid `border-t`/`border-b` slot triggers when they reintroduce extra padding.
- 2026-03-04: In dense settings side panels, avoid redundant nested section headers (for example duplicated provider card titles) and anchor tests to stable structure/ARIA labels instead of removable text labels.
- 2026-03-04: For UX-visible context captures (highlight -> follow-up), keep user-facing prompts human-readable (context blocks) and perform delimiter/XML wrapping only in backend prompt assembly so editing stays clean while model input stays structured.
- 2026-03-04: In floating settings panels, avoid modal interaction locks for inline selectors unless truly needed; prefer a non-locking combobox/select pattern and validate dismiss/select flows with Playwright.
- 2026-03-04: Performance optimizations for drag/resize must preserve live visual feedback; use frame-batched transient updates during interaction and defer expensive recomputation/persistence to interaction end.
- 2026-03-05: For user-selectable node colors, derive node-local semantic theme tokens from parsed color + contrast calculations (rather than styling only prompt/body text) so headers, markdown content, controls, borders, and icons all remain readable together.
- 2026-03-05: Workspace identity elements (chat title) should live at top-left with minimal/transparent chrome unless explicitly requested otherwise; avoid chip-like containers that compete visually with canvas content.
- 2026-03-05: For graph UX parity, use `onConnectEnd` + final connection state to support edge-drag into empty canvas creation; relying on `onConnect` alone only handles existing-node targets.
- 2026-03-05: Keep follow-up context as structured node metadata (rendered as dedicated context UI) and inject markup/XML only during backend prompt composition, never inside the editable prompt textarea.
- 2026-03-05: Context UI for follow-up nodes must persist in read mode as well as edit mode; users need to see attached context without entering prompt edit state.
- 2026-03-05: On Next.js 16+, `next lint` is no longer a valid CLI command; migrate to ESLint v9 flat config (`eslint.config.*`) and run `eslint .` in scripts.
